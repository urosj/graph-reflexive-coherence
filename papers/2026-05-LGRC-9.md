# Lorentzian Graph Reflexive Coherence (LGRC): Event-Driven RC with Emergent Causal Time

Copyright © 2026 Uroš Jovanovič, CC BY-SA 4.0.

This document defines the **Lorentzian / event-driven branch** of Graph
Reflexive Coherence. The existing GRC family evolves graph states through
globally ordered spatial slices. LGRC instead treats graph evolution as a
causal history: nodes carry local proper time, edges carry causal delay, and
updates occur when influence has arrived or a local proper-time threshold has
been reached.

The reflexive loop remains the same:

$$
C \;\rightarrow\; K[C] \;\rightarrow\; w[K] \;\rightarrow\; \Phi[w,C]
\;\rightarrow\; J[w,\Phi] \;\rightarrow\; \mathrm{continuity}
\;\rightarrow\; C.
$$

The new commitment is temporal:

$$
\text{scheduler order} \neq \text{proper time} \neq \text{checkpoint order}.
$$

The scheduler orders computation. Proper time orders influence. Checkpoints
are observer/replay artifacts.

LGRC is not a replacement for GRC. It is a conservative extension. Ordinary
synchronous GRC is recovered when lapse is uniform, edge delays are constant,
all nodes are eligible together, and delayed influence is not retained between
steps.

## Abstract

Graph Reflexive Coherence currently has a well-defined synchronous graph
family: GRC, GRC-9, and GRC9V3. These systems use globally indexed graph
states and are sufficient for coherence-induced geometry, flux, conservation,
nine-port mechanics, sparks, refinement, and basin-level identity witnesses.
Their update index, however, is an implementation scheduler. It is not yet an
emergent temporal geometry.

We define **Lorentzian Graph Reflexive Coherence (LGRC)** as the parallel
causal-history family. An LGRC state augments the graph substrate with local
proper times $\tau_i$, edge causal delays $\tau_{ij}$, lapse factors $N_i$,
chart-transport / lineage maps $\beta$, causal frontiers, and an event queue
$Q$. The spatial geometry of GRC is preserved, but influence is applied only
when it is causally available. Coherence conservation is extended from node
mass to node mass plus in-flight or pending coherence. Identity persistence is
measured over proper-time windows rather than raw step windows.

The paper gives the core LGRC state, temporal geometry, causal flux rules,
event-driven update contract, reduction to ordinary GRC, and specializations
to LGRC-9 and LGRC9V3. It also states implementation levels LGRC-0 through
LGRC-3 so that causal annotation, local proper-time eligibility, event queues,
and topology-changing causal histories are not confused.

## 0. Document Identity

This paper defines the **LGRC-9 family**. It introduces the minimal LGRC
causal runtime substrate needed for the nine-port specialization, but it does
not attempt a separate general LGRC or LGRC-V3 family paper.

It sits beside the existing synchronous family and should be read in that
lineage:

| Family | Runtime interpretation |
|---|---|
| GRC | Base synchronous graph Reflexive Coherence. |
| GRC-V2 | Synchronous graph RC with explicit graph substrate, sinks, basins, and topology-change mechanics. |
| GRC-V3 | Synchronous basin-attribute graph RC with node-local differential summaries, multi-metric edge labels, choice/collapse, and quadrature conservation. |
| GRC-9 | Synchronous nine-port graph RC. |
| GRC9V3 | Synchronous nine-port v3 basin/Hessian/spark runtime. |
| LGRC-9 | Lorentzian / event-driven nine-port graph RC. |
| LGRC9V3 | Lorentzian / event-driven nine-port v3 basin/Hessian/spark runtime. |
| LGRC / LGRC-V3 | Derivable future families, not separately defined as papers here. |

Shared concepts include coherence density, coherence-induced geometry,
conductance, potential, flux, conservation, sinks, basins, sparks, mechanical
refinement, budget preservation, and, in the nine-port families, ports, rows,
columns, column-H, and column G/Split.

The runtime semantics differ:

| Synchronous GRC | LGRC |
|---|---|
| Globally ordered spatial slices. | Causal histories. |
| Previous diagnostic usually means previous global step. | Previous diagnostic means previous causal evaluation of the relevant node or chart. |
| Step windows measure persistence. | Proper-time windows measure persistence. |
| Temporal delay may be an analytic edge label. | Temporal delay participates in causal arrival. |
| Checkpoints are often treated as state slices. | Checkpoints are explicitly observer/replay surfaces. |

This distinction matters especially for GRC9V3-style sparks. In synchronous
GRC9V3, a column-H sign crossing compares current values to the previous
global step. In LGRC9V3, the corresponding comparison must use the previous
causal evaluation event for that node or transported chart.

LGRC uses "Lorentzian" in the operational causal-structure sense:

```text
local proper time,
edge causal delay,
lapse,
causal cones,
event ordering,
and causal-history persistence.
```

This paper does not yet require a fully algebraic discrete Lorentzian metric
tensor or a signed interval form. Such a metric may be added later. The present
paper defines the causal runtime substrate that a later algebraic interval
theory would have to respect.

In short:

```text
In GRC, delay may be measured.
In LGRC, delay acts.
```

## 0.1 Non-Goals

This paper does not replace synchronous GRC. It defines a parallel
Lorentzian/event-driven family.

It does not eliminate computational ordering. It separates computational
ordering from physical or effective proper time.

It does not fix a final lapse formula, final edge-delay formula, or final
packet-transport policy across topology changes.

It does not require a fully algebraic discrete Lorentzian metric tensor.
"Lorentzian" is used here in the operational causal-structure sense described
above.

It does not make mechanical refinement into identity fission. Identity remains
basin-level persistence, now measured over an explicitly chosen proper-time
window.

It does not implement LGRC9V3 inside the base paper. LGRC9V3 is a later
runtime specialization that should reuse pure GRC9V3 formulas while replacing
the synchronous step loop with causal-history machinery.

## 0.2 GRC / LGRC Comparison

| Concept | GRC | LGRC |
|---|---|---|
| Evolution unit | Global step / spatial slice. | Causal event / local eligibility. |
| Time interpretation | Scheduler often doubles as update time. | Scheduler is distinct from proper time. |
| Delay | Analytic label or diagnostic. | Causal arrival condition. |
| Flux | Applied in slice update. | Packetized or delayed until arrival. |
| Persistence | Step window. | Proper-time window. |
| Previous diagnostic | Previous step. | Previous causal evaluation. |
| Refinement | Before/after slice. | Causal lineage event. |
| Checkpoint | State slice. | Observer/replay surface. |

## 1. Motivation

The existing GRC family uses indexed graph states. A typical synchronous
update has the form

$$
S^{(k)}
\;\longrightarrow\;
\text{geometry}
\;\longrightarrow\;
\text{potential}
\;\longrightarrow\;
\text{flux}
\;\longrightarrow\;
\text{continuity}
\;\longrightarrow\;
S^{(k+1)}.
$$

For node coherence this is often written schematically as

$$
C_i^{(k+1)}
=
C_i^{(k)}
-\Delta t \sum_{j\in\mathcal N(i)} J_{ij}^{(k)}.
\tag{1}
$$

This update is useful, auditable, and computationally stable. It gives event
payloads a clear state epoch, makes before/after topology comparisons easy,
and supports deterministic replay.

But the step index $k$ is an external scheduler. It tells the implementation
which state snapshot comes next. It does not, by itself, define effective time
inside the reflexive field.

In Reflexive Coherence, time is better understood as the ordered succession of
reflexive closure:

$$
C
\rightarrow
K[C]
\rightarrow
\text{geometry}
\rightarrow
J
\rightarrow
\text{continuity}
\rightarrow
C.
$$

The system does not merely evolve *in* time. Its local ordering is tied to how
coherence writes geometry, how geometry shapes flux, and how flux changes
coherence.

LGRC introduces graph mechanics for that distinction. It keeps computation
ordered, but refuses to identify computation order with physical or effective
time.

## 2. Minimal Runtime Contract And Invariants

LGRC should be specified as a causal-history runtime, not as a synchronous
runtime with extra labels.

The minimal contract is:

$$
\kappa \text{ orders computation},\qquad
\tau \text{ orders influence},\qquad
k \text{ orders snapshots}.
$$

Here $\kappa$ is the scheduler event index, $\tau$ denotes local proper time or
causal arrival time, and $k$ is an optional checkpoint index. These orders may
coincide in the synchronous limit, but they are not identified in LGRC by
default.

Every LGRC implementation level should preserve the following invariants.

| Invariant | Meaning |
|---|---|
| Monotone local proper time | A surviving node or transported chart lineage must not move backward in its own proper time. |
| Positive causal delay | Edge delay satisfies $\tau_{ij}>0$ unless a separately named degenerate limit is being studied. |
| Deterministic scheduler order | The scheduler may be artificial, but it must be deterministic and auditable for replay. |
| Snapshot non-ontology | A checkpoint index is not a physical simultaneity slice unless the synchronous limit is explicitly imposed. |
| Causal availability | A node update may use only local state and influence that has causally arrived. |
| Budget preservation | Asynchronous ordering must not create or destroy coherence. Delayed flux must be represented as in-flight or pending coherence. |
| Lineage-preserving refinement | Topology changes must transport node, port, basin, and event lineage. |
| Proper-time identity windows | Identity persistence is measured over local or lineage proper time, not over raw scheduler counts. |
| Fairness / non-starvation | If a node remains causally active and its lapse stays positive, its update eligibility must eventually be considered unless explicitly blocked by causal disconnection or policy. |

The conservation invariant for packet-based LGRC is

$$
B
=
\sum_{i\in V} C_i
+
\sum_{p\in P_{\mathrm{flight}}} C_p,
\tag{2}
$$

where $P_{\mathrm{flight}}$ is the set of in-flight coherence packets. A
delayed-evaluation implementation must provide an equivalent pending-flux
accounting surface.

These invariants are not optional engineering details. Without them, an
event-driven update can look Lorentzian while silently using the scheduler as
time, leaking budget, or treating checkpoint windows as identity evidence.

## 3. Three Notions Of Time

### 3.1 Scheduler Index

Let $\kappa\in\mathbb N$ denote the scheduler or event-order index. It answers:

$$
\text{Which computational event is processed next?}
$$

It does not answer:

$$
\text{Which event is earlier in the emergent causal/proper-time order?}
$$

The scheduler is still necessary. A finite implementation must choose an
event from an event queue, resolve ties, mutate state, and emit telemetry. The
claim of LGRC is not that computation has no ordering. The claim is that this
ordering is not automatically the geometry of time.

### 3.2 Snapshot Index

Let $k$ denote an optional checkpoint or replay index. A snapshot groups some
runtime state for inspection:

$$
\widehat S^{(k)} =
\text{observer-selected view of the causal history}.
$$

A snapshot can be emitted every $m$ scheduler events, at fixed wall-clock
intervals, at causal-frontier milestones, or on specific events such as sparks
and refinements.

In LGRC, a checkpoint is not a simultaneity slice unless the implementation
explicitly imposes a synchronous limit.

**Cross-paper notation note.** In synchronous GRC, GRC-9, and GRC9V3 papers,
$k$ is usually the primary step index. In this LGRC paper, the primary runtime
order is $\kappa$, while $k$ is reserved for observer snapshots. When comparing
LGRC formulas to synchronous formulas, read

$$
k_{\mathrm{GRC}}
\quad\leftrightarrow\quad
\kappa_{\mathrm{LGRC}}
$$

only in the synchronous limit where scheduler events and checkpoints coincide.

### 3.3 Proper Time And Causal Delay

Each node carries local proper time

$$
\tau_i(\kappa).
$$

Each edge or directed edge-channel carries causal delay

$$
\tau_{ij}(\kappa) > 0.
$$

A runtime may choose the source node's proper time as the departure event key,
but this is a policy choice. The general arrival expression is defined below
using event-time keys.

When delays are directed, $\tau_{i\to j}$ and $\tau_{j\to i}$ may differ. When
delays are symmetric, both directions share a single $\tau_{ij}$. LGRC should
make this policy explicit because causal reachability is different in the two
cases.

### 3.4 Timing Coordinate Note

This paper uses three related but distinct quantities:

| Quantity | Meaning |
|---|---|
| $\tau_i$ | Node-local proper time. |
| $\tau_{ij}$ | Edge causal delay. |
| $T_e$ | Event-time key used by the scheduler/event queue. |

An arrival expression should therefore be read as

$$
T_{\mathrm{arrive}}(i\to j)
=
T_{\mathrm{depart}}(i)+\tau_{ij}.
\tag{3}
$$

The departure term is the event-time key associated with the departure event,
not necessarily a claim that node $i$ and node $j$ share a private clock. A
runtime may set $T_{\mathrm{depart}}=\tau_i$ at departure, but this is a policy
choice and must be serialized.

When the arrival event is processed, node $j$'s local proper time is advanced
according to the chosen lapse/update policy. The arrival key and the target
node's accumulated proper time are related, but they are not identical by
definition.

## 4. LGRC State Object

A minimal LGRC state has the form

$$
S^{(\kappa)}
=
\bigl(
\Gamma,\; C,\; w,\; J,\; h,\; \tau,\; N,\; \beta,\; Q,\; P
\bigr),
\tag{4}
$$

where:

| Object | Meaning |
|---|---|
| $\Gamma$ | Graph topology, including port labels in nine-port families. |
| $C_i$ | Coherence density at node $i$. |
| $w_{ij}$ | Conductance / spatial metric surrogate. |
| $J_{ij}$ | Oriented coherence flux or flux proposal. |
| $h_{ij}$ | Spatial graph length or metric label. |
| $\tau_i$ | Local node proper time. |
| $\tau_{ij}$ | Edge causal delay. |
| $N_i$ | Local lapse, the rate at which proper time accumulates relative to scheduler advancement. |
| $\beta$ | Chart transport / lineage map across causal updates and topology changes. |
| $Q$ | Event queue or causal frontier. |
| $P$ | In-flight or pending coherence records, if the flux policy delays application. |

An LGRC event should minimally contain:

| Field | Meaning |
|---|---|
| `event_id` | Stable event identifier. |
| `scheduler_event_index` | Computational order $\kappa$. |
| `event_kind` | Arrival, local update, spark candidate, refinement, collapse, checkpoint, etc. |
| `node_id` / `edge_id` | Local target, source, or channel. |
| `node_proper_time` | Relevant local proper time. |
| `causal_epoch` | Pre-update, post-arrival, pre-refinement, post-refinement, etc. |
| `payload` | Diagnostic and transition evidence. |

This state object is a graph-history state. It is not merely a decorated
spatial slice.

## 5. Spatial Geometry Layer

LGRC inherits the spatial geometry layer from GRC.

Let $h_{ij}$ be a spatial edge length or metric label associated with edge
$\{i,j\}$. A common policy is

$$
h_{ij}=f(w_{ij}),
\tag{5}
$$

for example

$$
h_{ij}=-\log(w_{ij}+\epsilon).
\tag{6}
$$

The exact policy is family- and implementation-dependent. LGRC requires only
that spatial geometry remain auditable and that temporal delay not be confused
with spatial length.

**Cross-paper notation note.** GRC-V3 uses $\ell_{ij}^{(d)}$ for geometric
length. This paper writes the same role more generically as $h_{ij}$:

$$
h_{ij}
\;\equiv\;
\ell_{ij}^{(d)}
\quad\text{when using the GRC-V3 edge-label convention}.
$$

In GRC9V3, edge labels such as geometric length, temporal delay, and flux
coupling may already be present as analytic labels. In synchronous GRC they are
labels. In LGRC, the temporal label becomes operational when it participates
in causal arrival.

## 6. Temporal Geometry Layer

LGRC adds temporal geometry through edge delay and node lapse.

### 6.1 Edge Delay

For each edge or directed edge-channel define

$$
\tau_{ij}
=
T(C_i,C_j,w_{ij},J_{ij},h_{ij},\chi_{ij}),
\tag{7}
$$

where $\chi_{ij}$ may include capacity, curvature, saturation, basin status,
or family-specific analytic labels.

This paper does not fix a final formula for $T$. It fixes the required
properties:

1. $\tau_{ij}>0$.
2. $\tau_{ij}$ is auditable from runtime state or event payloads.
3. $\tau_{ij}$ may depend on coherence-induced geometry.
4. $\tau_{ij}$ admits a constant-delay specialization in the synchronous GRC limit.
5. Directed delay, if used, must be explicitly named.

A simple first implementation can set

$$
\tau_{ij}=\tau_0\,g(h_{ij}),
\qquad g(h)>0,
\tag{8}
$$

and later add flux, capacity, or curvature dependence.

The GRC-V3 temporal label already provides a useful flux-dependent candidate:

$$
\tau_{ij}
=
\frac{\ell_{ij}^{(d)}}{v_0+\rho |J_{ij}|+\varepsilon_\tau}.
\tag{8a}
$$

In synchronous GRC-V3 this is an analytic propagation label. In LGRC it can be
made operational by using it as the causal arrival delay. Equation (8) is
therefore only the simplest first policy, not the only natural starting point.

### 6.2 Lapse

The lapse $N_i>0$ controls local proper-time accumulation relative to scheduler
advancement:

$$
\tau_i(\kappa+1)
=
\tau_i(\kappa)+N_i(\kappa)\Delta\kappa.
\tag{9}
$$

If $N_i$ is arbitrary, LGRC has merely assigned clocks. To be reflexive, $N_i$
should be derived from local state:

$$
N_i
=
\mathcal N(C_i,K_i,J_i,\deg_{\mathrm{act}}(i),\text{basin}_i,\ldots).
\tag{10}
$$

The exact lapse policy can remain open in the first paper, but its role should
not. Lapse is where coherence state affects temporal geometry, temporal
geometry affects update eligibility, and update eligibility changes coherence
state.

There are two basic lapse-accumulation policies:

| Policy | Meaning |
|---|---|
| Global scheduler accumulation | Every scheduler event advances every node's proper time by its configured lapse rule. |
| Local event/frontier accumulation | A node's proper time advances only when processing an event on that node or when an explicit causal-frontier update advances it. |

LGRC must choose and serialize the policy. LGRC-1 may use global scheduler
accumulation for simplicity. LGRC-2 should prefer event/frontier-based
accumulation once causal arrivals and event queues are operational.

In event-driven accumulation, a node's proper time need not be eagerly updated
on every unrelated scheduler event. It may advance **virtually** through the
event key used to schedule that node's next local update or arrival. When the
event is processed, the runtime materializes the elapsed proper time using the
serialized lapse policy and the previous proper-time timestamp for that node or
chart lineage.

The policy also interacts with fairness. A positive lapse should not allow a
causally active node to be starved indefinitely by unrelated rapid event
cycling elsewhere in the graph.

## 7. Shift As Chart Transport

In continuum Lorentzian geometry, shift describes how spatial coordinates
slide between slices. On a graph, the corresponding object is not naturally a
vector. LGRC uses a chart-transport / lineage map

$$
\beta.
$$

The map $\beta$ answers:

$$
\text{Which later local chart continues which earlier local chart?}
$$

When topology is unchanged, $\beta$ may be identity. During refinement, it is
nontrivial. For a nine-port expansion,

$$
\text{parent node}
\;\longrightarrow\;
\text{core + satellites},
$$

and $\beta$ must record at least:

1. parent node to module lineage,
2. old parent port to new endpoint lineage,
3. old column to new column-family lineage,
4. parent basin to candidate child-basin lineage,
5. causal event lineage from candidate to expansion.

This is the graph analogue of shift that matters for RC: not coordinate
sliding as a vector field, but lineage-preserving chart transport across
causal updates.

## 8. Causal Cones

Adjacency is not causal reachability.

Given edge delays, the delay of a path

$$
p=(i=i_0,i_1,\ldots,i_m=j)
$$

is

$$
\tau_{\mathrm{path}}(p)
=
\sum_{\ell=0}^{m-1}\tau_{i_\ell i_{\ell+1}}.
\tag{11}
$$

The future cone of node $i$ over proper-time budget $\Delta\tau$ is

$$
\mathrm{Future}_i(\Delta\tau)
=
\{\,j\mid \exists p:i\to j,\ \tau_{\mathrm{path}}(p)\le\Delta\tau\,\}.
\tag{12}
$$

The past cone is defined analogously using incoming or reversed causal
channels:

$$
\mathrm{Past}_i(\Delta\tau)
=
\{\,j\mid \exists p:j\to i,\ \tau_{\mathrm{path}}(p)\le\Delta\tau\,\}.
\tag{13}
$$

In synchronous GRC, all adjacent influence can be treated as available at the
next update. In LGRC, an adjacent edge is only a channel. Influence must still
arrive.

A causal frontier is not necessarily a set of mutually simultaneous events. It
is an observer- or scheduler-selected boundary in the causal history. Different
frontier policies may expose different but equivalent replay surfaces, and the
policy must be serialized when it is used for telemetry, visualization, or
identity evidence.

This gives a direct operational distinction between:

| Graph fact | Causal fact |
|---|---|
| $j\in\mathcal N(i)$ | There is an available edge-channel. |
| $j\in\mathrm{Past}_i(\Delta\tau)$ | Influence from $j$ could have reached $i$ within the proper-time budget. |

## 9. Three Distances In LGRC

LGRC preserves the GRC distinction between three edge/path notions:

1. geometric separation,
2. temporal / causal delay,
3. functional coupling.

It changes the operational status of the temporal notion. In synchronous GRC,
temporal delay can be stored as an analytic edge label. In LGRC, temporal
delay participates in causal arrival and update eligibility.

For a path

$$
p=(i_0,i_1,\ldots,i_m),
$$

define the geometric path distance as

$$
D_h(p)
=
\sum_{\ell=0}^{m-1} h_{i_\ell i_{\ell+1}},
\tag{14}
$$

the causal / proper-time path delay as

$$
D_\tau(p)
=
\sum_{\ell=0}^{m-1} \tau_{i_\ell i_{\ell+1}},
\tag{15}
$$

and a functional path score as a policy-dependent coupling functional

$$
D_\chi(p)
=
\mathcal D_\chi
\bigl(
w,\;J,\;\text{flux coupling},\;\text{capacity},\ldots
\bigr).
\tag{16}
$$

The functional quantity may be represented as a cost, a strength, or a
max-capacity criterion depending on the analysis. The important point is that
it is not automatically the same as geometric length or causal delay.

Thus, a path may be:

```text
geometrically short but causally slow;
causally fast but functionally weak;
functionally dominant but geometrically indirect.
```

LGRC should keep these surfaces separate in telemetry and visualization:

| Surface | Meaning | LGRC status |
|---|---|---|
| $h_{ij}$ or $\ell_{ij}^{(d)}$ | Geometric / spatial separation. | Preserved from GRC. |
| $\tau_{ij}$ | Temporal / causal delay. | Upgraded from label to operational arrival delay. |
| coupling / flux / capacity score | Functional transfer relation. | Preserved as dynamical/effective connectivity. |

This is why causal cones are not merely shortest-path neighborhoods and why
proper-time path disagreement remains meaningful in LGRC.

## 10. Causal Flux And Conservation

A Lorentzian/event-driven graph runtime must define how flux respects causal
delay. There are two natural implementation strategies.

LGRC distinguishes four flux-related quantities:

| Quantity | Meaning |
|---|---|
| $J_{\mathrm{proposal}}$ | Local flux proposal computed from currently causally available state. |
| $C_p$ or `packet_amount` | Coherence amount actually placed in flight. |
| $J_{\mathrm{arrived}}$ | Arrived contribution available to the target node. |
| $J_{\mathrm{applied}}$ | Contribution used in a local continuity update. |

Only packet departure/arrival, or an equivalent pending-flux accounting
surface, changes the conserved budget. A flux proposal alone is diagnostic
until a runtime turns it into a packet, pending contribution, or applied
continuity term.

### 10.1 Flux Packets

Flux leaving $i$ along edge $ij$ at departure time $\tau_i^{\mathrm{depart}}$
is represented by a packet

$$
p =
\bigl(
i,\;j,\;e,\;C_p,\;\tau_{\mathrm{depart}},\;\tau_{\mathrm{arrive}},\;\ell_p
\bigr),
\tag{17}
$$

where $\ell_p$ is a causal lineage id and

$$
\tau_{\mathrm{arrive}}
=
\tau_{\mathrm{depart}}+\tau_{ij}.
\tag{18}
$$

Departure and arrival update the budget as:

$$
\begin{aligned}
\text{departure:}\quad
&C_i \leftarrow C_i-C_p,\qquad
P_{\mathrm{flight}}\leftarrow P_{\mathrm{flight}}\cup\{p\},\\
\text{arrival:}\quad
&C_j \leftarrow C_j+C_p,\qquad
P_{\mathrm{flight}}\leftarrow P_{\mathrm{flight}}\setminus\{p\}.
\end{aligned}
\tag{19}
$$

Then the invariant (2) is exact.

### 10.2 Delayed Flux Evaluation

Alternatively, the runtime may compute flux proposals but apply them only
when arrival conditions are satisfied. In that case, pending contributions
must be recorded explicitly. A hidden pending term is not acceptable because
it makes conservation impossible to audit.

The difference from packetized flux is representational, not conceptual.

| Strategy | Representation | Conservation surface |
|---|---|---|
| Flux packets | Each departing contribution is an explicit packet with source, target, amount, departure time, arrival time, and lineage. | Node coherence plus packet coherence. |
| Delayed evaluation | Contributions are stored in a pending accumulator or delay buffer, often aggregated by target, edge, or arrival epoch. | Node coherence plus pending-buffer coherence. |

Packets are the most explicit and auditable form. Delayed evaluation is a more
compact implementation of the same causal idea, but it must still expose enough
pending-flux state to prove budget preservation.

### 10.3 Conservation Requirement

LGRC must preserve coherence under asynchronous order. The minimum acceptable
budget audit is

$$
B^{(\kappa)}
=
\sum_i C_i^{(\kappa)}
+
\sum_{p\in P^{(\kappa)}} C_p,
\qquad
B^{(\kappa+1)}=B^{(\kappa)}
\tag{20}
$$

up to stated numerical tolerance.

This is the most important technical condition in the paper. Without it,
asynchronous dynamics can create artifacts that look like causal time but are
only update-order leakage.

## 11. Event-Driven Update Rule

For delayed-evaluation LGRC, a local continuity-style update may use local
proper time and causally available influence:

$$
C_i(\tau_i+\Delta\tau_i)
=
C_i(\tau_i)
-
\Delta\tau_i
\sum_{j\in\mathcal A_i(\tau_i)} J_{ij},
\tag{21}
$$

where $\mathcal A_i(\tau_i)$ is the set of neighbors or incoming packets whose
influence has arrived for node $i$ at local proper time $\tau_i$.

Here

$$
\Delta\tau_i
=
\tau_i-\tau_i^{\mathrm{last\_update}}
$$

is the elapsed local proper time since node $i$'s previous update.

For packetized LGRC, the equivalent coherence movement occurs through packet
departure and arrival events:

```text
departure:
    source node coherence decreases;
    in-flight coherence increases.

arrival:
    in-flight coherence decreases;
    target node coherence increases.
```

The delayed-evaluation update (21) and packet departure/arrival updates should
not be applied simultaneously unless a runtime defines an explicit equivalence
rule. Otherwise the same flux can be counted twice.

A simple local eligibility rule is

$$
\tau_i-\tau_i^{\mathrm{last\_update}}\ge \theta_i,
\tag{22}
$$

where $\theta_i>0$ is a local update threshold.

The event queue contains at least:

1. causal arrivals,
2. local update threshold crossings,
3. topology events such as spark candidates and expansions,
4. observer events such as checkpoints.

The next event is chosen by earliest causal/scheduler key with deterministic
tie-breaking:

$$
e_{\mathrm{next}}
=
\arg\min_{e\in Q}
\bigl(
\tau_{\mathrm{event}}(e),\;
\mathrm{priority}(e),\;
\mathrm{event\_id}(e)
\bigr).
\tag{23}
$$

The exact tie-break policy is implementation-specific, but it must be
deterministic and serialized.

The event-time key must be defined per event kind. A minimal policy is:

| Event kind | $\tau_{\mathrm{event}}(e)$ |
|---|---|
| Causal arrival | Arrival time $\tau_{\mathrm{arrive}}$. |
| Local update eligibility | Node proper time at threshold crossing. |
| Spark candidate | Proper time of the candidate node at diagnostic evaluation. |
| Mechanical refinement | Proper time inherited from the triggering candidate event, plus any explicit refinement delay. |
| Identity persistence check | End of the configured proper-time persistence window. |
| Checkpoint / observer event | Observer-selected sampling key; not physical simultaneity unless synchronous limit is imposed. |

If several event kinds share the same event time, the serialized priority and
event id determine processing order.

## 12. Implementation Levels

The theory should not require all causal machinery in the first runtime.
Instead, LGRC should be implemented in levels. Each level preserves the
invariants of Section 2; the difference is how much of the causal layer is
operational rather than derived.

### 12.1 LGRC-0: Causal Annotation

LGRC-0 changes no behavior. It runs an ordinary GRC or GRC9V3 model and
derives:

$$
\tau_i,\quad
\tau_{ij},\quad
N_i,\quad
\mathrm{Past/Future}_i(\Delta\tau),\quad
\tau_{\mathrm{path}}.
$$

This is useful for diagnostics, visualization, and calibration. It must not
claim event-driven causal update semantics.

### 12.2 LGRC-1: Local Proper-Time Eligibility

LGRC-1 keeps a global scheduler but lets nodes update only when local
proper-time eligibility holds:

$$
\tau_i\leftarrow\tau_i+N_i\Delta\kappa,
\qquad
\tau_i-\tau_i^{\mathrm{last\_update}}\ge\theta_i.
\tag{24}
$$

This is the first behavioral LGRC level. It should be accompanied by explicit
budget audits showing that skipped or delayed local updates do not leak
coherence.

LGRC-1 is a transitional runtime. It introduces local proper-time eligibility,
but it may still use the latest available neighbor state rather than full
causal packet arrival. Unless it also introduces explicit causal availability
buffers, it should be labeled **semi-causal** rather than a complete
event-driven Lorentzian runtime.

### 12.3 LGRC-2: Event-Driven Causal Queue

LGRC-2 uses an event queue of arrivals and local update thresholds:

$$
\text{process earliest event}
\;\rightarrow\;
\text{update affected local state}
\;\rightarrow\;
\text{schedule future arrivals}.
$$

At this level, in-flight or pending coherence accounting is mandatory.

### 12.4 LGRC-3: Topology-Changing Causal History

LGRC-3 adds sparks, refinements, expansions, identity persistence, collapse,
and reabsorption to the causal event history.

At this level, coherence transfer and time/lineage transfer are separate
contracts. A refinement that preserves budget but loses causal lineage is not
a valid LGRC identity event.

## 13. Sparks In LGRC

In synchronous GRC, a spark candidate is detected at a step and state epoch.
In LGRC, a spark candidate is detected at a causal event:

$$
e_s
=
(\kappa_s,\;s,\;\tau_s,\;\mathrm{causal\_epoch},\;\mathrm{payload}).
\tag{25}
$$

The spark check is not run merely because the scheduler advances. It is
triggered by causal events that can change the local diagnostic state, such as:

1. arrival of influence or pending flux at the candidate node;
2. a local update threshold crossing;
3. topology or lineage transport involving the node or its local chart;
4. an explicit diagnostic reevaluation event scheduled by the runtime.

Existing diagnostics can still be used:

| Synchronous diagnostic | LGRC interpretation |
|---|---|
| active degree | Evaluated at the node causal event. |
| sink status | Evaluated in the causal state available to the node. |
| signed Hessian | Evaluated at the node/basin causal epoch. |
| column-H | Evaluated from causally available neighbor state. |
| previous column-H | Previous causal evaluation of this node/chart. |

If the node or chart has undergone refinement, replacement, or lineage
transport, previous diagnostics must be resolved through $\beta$. If the
transport map cannot identify a valid predecessor chart, the previous
diagnostic is unavailable rather than guessed.

A candidate payload should include:

1. event id,
2. scheduler event index,
3. node id,
4. node proper time,
5. causal epoch,
6. active degree and sink/basin status,
7. diagnostic values,
8. causal cone or arrival context where relevant,
9. gate reasons.

For LGRC9V3, the Lane B distinction remains meaningful: a column-H branch is
direct event evidence only when that branch is part of the selected LGRC9V3
spark policy. The fact that the diagnostic is causal-time-aware does not
promote the column-H proxy into a true geometric Hessian.

## 14. Refinement In LGRC

Mechanical refinement in GRC-9 replaces a saturated sink with a module and
reassigns old boundary edges by column. LGRC preserves the mechanical idea but
adds causal-time obligations.

When node $s$ expands into module $M_s$, the runtime must define:

1. coherence transfer from $s$ to module nodes,
2. proper-time inheritance from $s$ to module nodes,
3. causal delay of new internal edges,
4. causal status of reassigned boundary edges,
5. handling of in-flight packets involving $s$,
6. lineage transport from parent chart to child charts.

State transfer and time transfer are different. A rule such as

$$
C_s \longrightarrow (C_{s_0},C_{s_1},C_{s_2},C_{s_3})
\tag{26}
$$

does not automatically define

$$
\tau_s \longrightarrow
(\tau_{s_0},\tau_{s_1},\tau_{s_2},\tau_{s_3}).
\tag{27}
$$

Possible time-inheritance policies include:

| Policy | Meaning |
|---|---|
| Uniform inheritance | All module nodes inherit parent proper time. |
| Core-first inheritance | Core inherits parent proper time; satellites become causally separated through internal delays. |
| Column-offset inheritance | Satellites inherit parent proper time plus column-specific offsets. |

The first LGRC9V3 implementation should choose the simplest auditable policy,
probably uniform inheritance plus explicit internal edge delays. More complex
policies should be introduced only when experiments create tension.

In-flight packet handling is the urgent topology-change case. A refinement
must define what happens to packets whose source, target, or traversed edge
references the expanded node $s$.

These packet-refinement policies are required once a runtime supports
in-flight coherence packets. They are not required for LGRC-0 annotation-only
or for an LGRC-1 implementation that has not introduced packetized causal
flux.

| Packet relation to $s$ | Required refinement policy |
|---|---|
| Packet departing from $s$ but not yet arrived | Rebase source lineage to the appropriate module node, or preserve parent-source lineage until arrival with an explicit parent-to-module resolution rule. |
| Packet arriving at $s$ after expansion | Redirect target to the appropriate module endpoint, usually by old boundary column or lineage transport. |
| Packet crossing an old boundary edge | Reassign by the same old-port / old-column rule used for boundary edge reassignment. |
| Packet on an internal edge being replaced | Either deliver before refinement, cancel with budget-neutral rollback, or transport to a new internal edge with explicit lineage. |

The key invariant is:

$$
\text{packet amount before refinement}
=
\text{packet amount after refinement}
$$

unless an explicitly named arrival, rollback, or cancellation event accounts
for the difference. Mechanical expansion must not lose in-flight coherence.

## 15. Identity Persistence In LGRC

Identity remains basin persistence, not graph rewriting.

In synchronous GRC9V3, a clean fixture may say:

$$
\text{child basin persisted for } n \text{ steps}.
$$

In LGRC, the corresponding claim is:

$$
\text{child basin persisted for } \Delta\tau\ge\tau_{\mathrm{persist}}.
\tag{28}
$$

The identity window must specify which proper-time clock is used:

| Clock | Meaning |
|---|---|
| Sink-local proper time | Persistence measured by the child sink's clock. |
| Lineage proper time | Persistence measured along the transported parent-to-child lineage. |
| Basin aggregate proper time | Persistence measured by a minimum, average, or mass-weighted basin clock. |
| Causal-frontier time | Persistence measured by an observer/frontier policy. |

The first LGRC9V3 implementation should choose one clock policy and serialize
it in identity evidence. A claim of proper-time persistence is incomplete
without this choice.

Identity evidence requires:

1. mechanical refinement or another lineage-bearing transition,
2. child sink/basin evidence,
3. lineage from parent or module,
4. basin mass or stability threshold,
5. budget preservation,
6. proper-time persistence,
7. causal-history consistency.

Mechanical refinement is still not identity fission. A causal spark is still
not identity fission. Identity is accepted only when the post-event basin
structure persists in proper time.

## 16. Reduction To Ordinary GRC

LGRC must reduce to ordinary GRC under a uniform synchronous limit.

Assume:

1. $N_i=1$ for every node,
2. $\tau_{ij}=\tau_0$ for every active edge,
3. every node is eligible at each global update,
4. all edge influence available in the current slice is applied together,
5. no in-flight packet state is retained across updates.

Then

$$
\tau_i(\kappa)=\kappa\Delta t
\tag{29}
$$

for all nodes, and the local update (21) reduces to the synchronous update
(1):

$$
C_i^{(k+1)}
=
C_i^{(k)}
-\Delta t\sum_{j\in\mathcal N(i)}J_{ij}^{(k)}.
\tag{30}
$$

**Proposition.** Under the assumptions above, LGRC and synchronous GRC produce
the same coherence update for a fixed topology and the same flux law.

*Proof.* Uniform lapse gives a shared proper-time increment for all nodes.
Constant delay and synchronous eligibility make every adjacent influence
available at the same update boundary. Because no in-flight state is retained,
the causal availability set $\mathcal A_i$ equals the ordinary neighbor set
$\mathcal N(i)$ at each update. Substituting this into (21) gives (30). *QED*

This reduction is essential. LGRC extends GRC; it does not invalidate the
spatial-slice family.

It inherits from GRC-9:

2. the 3x3 row/column chart,

4. rows as local differential directions,
5. columns as interface/refinement/multiscale families,

LGRC-9 is not GRC-9 with an extra temporal label. It is nine-port graph RC on

## 20. Implementation Architecture

LGRC should not be hidden as a mode inside synchronous GRC runtimes.

Recommended separation:

```text

src/pygrc/models/lgrc_9_v3.py
```

or, if the package is later reorganized by family:

```text

src/pygrc/lgrc/
src/pygrc/lgrc9/
src/pygrc/lgrc9v3/
```

The engineering reason is straightforward: the synchronous runtime and the
Lorentzian runtime have different control flow. Shared formulas should be
factored out, but update logic should not be merged.

The first implementation should probably target:

1. LGRC-0 causal annotation on existing GRC9V3 outputs,
2. LGRC-1 local proper-time eligibility on fixed topology,
3. only then LGRC-2 causal flux packets,
4. only then LGRC-3 topology-changing LGRC9V3.

This order follows the GRC principle itself: do not create geometry changes
unless the current runtime has generated tension that requires them.

## 21. Experiment Program

LGRC should have its own experiment family. The experiments should separate
derived annotation from operational causal dynamics.

### LT-A: Causal Annotation

Given a synchronous GRC or GRC9V3 run, compute $\tau_i$, $\tau_{ij}$, $N_i$,
causal cones, and proper-time paths without changing behavior.

Purpose: validate deterministic reconstruction and visualization of the
Lorentzian layer.

### LT-B: Proper-Time Path Disagreement

Compare geometric shortest path, causal/proper-time shortest path, and
strongest functional flux/coupling path.

Purpose: show that causal/proper-time reachability is not reducible to spatial
graph reachability.

### LT-C: Local Lapse Effects

Show that nodes accumulate proper time at different rates under different
local coherence, geometry, or flux states.

### LT-D: Local Update Eligibility

Run LGRC-1 and verify that local proper-time eligibility changes update order
while preserving budget. If no causal availability buffers are present, record
the result as semi-causal LGRC-1 rather than full event-driven LGRC.

### LT-D2: Packet-Loop Implementation Example

A later LGRC9V3 implementation instantiates LGRC-2 packetized causal flux as
state-triggered self-rearming packet loops. That mechanism uses route-aspect
semantics, measured pole-surplus triggers, and native self-rearm evidence to
produce a D2.3-equivalent packet loop under controls.

This is an implementation example of LGRC-2 packetized causal flux, not part
of the minimal LGRC definition. The mechanism is documented separately in
`2026-05-LGRC9V3-Native-Packet-Loops.md`.

### LT-E: Causal Spark Ordering

Show that spark candidates can be ordered by causal/proper time rather than
global step index.

### LT-F: Identity Persistence In Proper Time

Replace step-window persistence with proper-time-window persistence and test
whether identity evidence changes.

### LT-G: Reduction Test

Verify that uniform lapse, constant delay, and synchronous eligibility
reproduce ordinary GRC.

## 22. Open Questions

This paper deliberately leaves several design questions open:

1. How should in-flight packets be transported across refinement?
2. How should proper time be inherited across refinement?
3. Should causal flux use explicit packets or compact delayed evaluation?
4. How should in-flight coherence be represented compactly?
5. What is the best formula for lapse $N_i$?
6. What is the best formula for edge delay $\tau_{ij}$?
7. Should delay be symmetric or directed?
8. How should proper-time identity thresholds be calibrated?
9. How should LGRC and GRC results be compared fairly?
10. Should Lorentzian signature be represented algebraically or operationally?
11. How do causal cones interact with basin identity?
12. How should collapse and reabsorption be represented in causal history?

Leaving these open is not a weakness. It keeps the first LGRC paper from
over-specifying runtime choices before experiments create the necessary
tension.

### 22.1 First-Round Default Policy Summary

The questions above are intentionally not closed by theory alone. However, the
first LGRC implementation should use conservative, auditable defaults so that
experiments have a stable target.

The first-round defaults are:

1. bounded density/tension lapse;
2. GRC-V3 temporal-delay label as operational edge delay, with a simpler
   geometry-only baseline for LGRC-0/LGRC-1;
3. symmetric delay before directed delay;
4. packetized flux for LGRC-2;
5. compact pending-flux ledgers, with lineage retained whenever topology
   changes are enabled;
6. lineage-preserving packet transport across refinement;
7. uniform proper-time inheritance under refinement;
8. identity persistence calibrated to local causal traversal time;
9. GRC/LGRC comparisons aligned by proper-time surfaces and event classes;
10. operational causal-cone Lorentzian structure rather than algebraic metric
   signature;
11. distinction between full flux basin and causal basin core;
12. collapse/reabsorption represented as causal lineage events with budget and
    proper-time transfer evidence.

These are first-round policies, not final theory. Experimental tension may
promote alternatives such as directed delay, column-offset time inheritance,
corroborated spark policies, compact delayed-evaluation optimizations, or
algebraic Lorentzian interval structures.

Details and formulas are recorded in Appendix G.

## 23. Claim Discipline

Safe claims:

1. LGRC separates scheduler order from emergent proper time.
2. LGRC defines causal cones and local proper-time accumulation on graph histories.
3. LGRC reduces to ordinary GRC under the uniform synchronous limit.
4. LGRC lets identity persistence be measured over proper-time windows.
5. LGRC can reuse GRC-9 and GRC9V3 spatial, port, and refinement concepts.

Unsafe claims:

1. LGRC removes computational ordering.
2. LGRC proves physical spacetime.
3. Every synchronous GRC result was wrong because it used steps.
4. Asynchronous updates automatically preserve budget.
5. Mechanical refinement is identity fission.
6. Proper-time persistence is established without causal-history evidence.

The central discipline is:

$$
\text{do not confuse scheduler order, checkpoint order, and proper time}.
$$

## 24. Closing Summary

LGRC is the Lorentzian/event-driven branch of Graph Reflexive Coherence.

It does not say synchronous GRC is wrong. It says synchronous GRC is the
spatial-slice implementation, while LGRC is the causal-history implementation.

The guiding distinction is:

$$
\text{GRC: graph coherence over ordered spatial slices}
$$

and

$$
\text{LGRC: graph coherence over event-driven causal histories}.
$$

The guiding sentence is:

$$
\text{The scheduler orders computation. Proper time orders influence.}
$$

LGRC therefore provides the missing bridge between Reflexive Coherence as a
field theory, where time is tied to reflexive closure, and graph
implementations where update order must be explicit, auditable, and eventually
causal.

## Appendix B. Causal Cone Algorithms

For nonnegative edge delays, shortest proper-time paths can be computed by
Dijkstra-style algorithms on the directed or undirected delay graph.

Future cone:

1. start at source node $i$ with accumulated delay $0$;
2. relax outgoing causal channels by $\tau_{uv}$;
3. retain nodes whose best accumulated delay is at most $\Delta\tau$.

Past cone:

1. reverse directed causal channels, or use incoming channels directly;
2. run the same bounded shortest-delay search;
3. retain nodes that can reach $i$ within $\Delta\tau$.

The algorithmic result is not merely a graph neighborhood. It is a
delay-bounded causal neighborhood.

## Appendix C. Conservation With Flux Packets

Packet departure subtracts coherence from a source node and adds the same
coherence to the in-flight packet set. Packet arrival subtracts the same
coherence from the in-flight set and adds it to the target node. Therefore

$$
\sum_i C_i+\sum_p C_p
$$

is invariant under both operations.

This simple proof is the main reason packetized flux is the cleanest first
model for LGRC-2, even if later runtimes choose a more compact delayed
evaluation representation.

## Appendix D. LGRC-9 Port Specialization

In LGRC-9, each causal event involving a port-labeled edge should record the
local endpoint port used by the causal source and target. For a port
$r\in\{1,\ldots,9\}$,

$$
a=\left\lfloor\frac{r-1}{3}\right\rfloor+1,
\qquad
b=((r-1)\bmod 3)+1.
$$

Rows remain local differential directions. Columns remain interface,
refinement, and multiscale families. Ports remain row-column intersections.

The difference is that every row, column, and port observation is now attached
to a causal event, not merely to a global state slice.

## Appendix G. First-Round LGRC Policy Defaults

This appendix records conservative first-round defaults for the open questions
in Section 22. They are chosen to be minimal, auditable, experimentally
falsifiable, and reducible to synchronous GRC under uniform conditions.

They are not final theory. Each default may be replaced by an explicitly named
policy when experiments require it.

### G.1 Bounded Density/Tension Lapse

First-round policy:

$$
N_i =
\mathrm{clip}\!\left(
1
+\lambda_N\frac{C_i-C_{\mathrm{ref}}}{C_{\mathrm{ref}}+\varepsilon_C}
+\mu_N\frac{\|\mathbf g_i\|}{g_{\mathrm{ref}}+\varepsilon_g},
N_{\min},
N_{\max}
\right),
\qquad
\lambda_N,\mu_N\ge 0.
$$

Here $C_{\mathrm{ref}}$ and $g_{\mathrm{ref}}$ may initially be global medians
for calibration, but should later be local causal-neighborhood references.
The small constants $\varepsilon_C,\varepsilon_g>0$ prevent division by zero.
In a uniform state with $C_i=C_{\mathrm{ref}}$ and $\|\mathbf g_i\|=0$, this
policy gives $N_i=1$ before clipping, preserving the synchronous-limit
normalization.

Interpretation:

```text
N_i is a local reflexive activity-rate policy,
not a literal GR gravitational lapse.
```

High coherence and high local tension cause more proper-time accumulation per
scheduler increment under this policy. The clipping bounds prevent extreme
local activity from dominating the event queue and violating the fairness /
non-starvation invariant.

Status:

```text
default for LGRC-1 experiments;
calibration open.
```

### G.2 Operational Edge Delay

First-round policy:

$$
\tau_{ij}
=
\max\!\left(
\tau_{\min},
\frac{\ell_{ij}^{(d)}}{v_0+\rho |J_{ij}|+\varepsilon_\tau}
\right).
$$

For LGRC-0/LGRC-1 annotation or semi-causal runs, a simpler baseline may be
used:

$$
\tau_{ij}=\tau_0\,\ell_{ij}^{(d)}.
$$

The GRC-V3 label becomes operational when used as the LGRC causal arrival
delay. When flux is negligible, the first formula reduces to a
geometry-proportional delay, not necessarily a constant unless geometric
lengths are normalized.

Status:

```text
default operational delay candidate;
simple geometry-only baseline allowed for LGRC-0/LGRC-1.
```

### G.3 Symmetric Delay First, Directed Delay Optional

First-round policy:

$$
\tau_{i\to j}=\tau_{j\to i}=\tau_{ij}.
$$

Directed delay is an optional tension-driven extension:

$$
\tau_{i\to j}
=
\mathrm{clip}\!\left(
\tau_{\mathrm{base},ij}
\exp\!\left(-\gamma\frac{J_{i\to j}}{J_{\mathrm{ref}}}\right),
\tau_{\min},
\tau_{\max}
\right).
$$

The sign convention must be explicit: $J_{i\to j}>0$ means flux favors the
$i\to j$ direction. The normalization $J_{\mathrm{ref}}$ and clipping bounds
are required for stability.

Status:

```text
symmetric delay is default;
directed delay remains optional at every LGRC level.
```

### G.4 Packetized Causal Flux

First-round LGRC-2 should use explicit packets:

```text
departure:
    node coherence decreases;
    in-flight coherence increases.

arrival:
    in-flight coherence decreases;
    target node coherence increases.
```

This directly preserves:

$$
B=\sum_i C_i+\sum_p C_p.
$$

Delayed evaluation may be introduced later as a compact representation, but
only if it exposes an equivalent pending-flux ledger.

Status:

```text
default for LGRC-2;
required before claiming full event-driven causal flux.
```

### G.5 Pending-Flux Ledger

For fixed-topology runs, in-flight coherence may be represented compactly by
an edge ledger:

$$
L_{ij}
=
\{\,(\tau_{\mathrm{arrive}}^{(m)},C_{\mathrm{pending}}^{(m)})\,\}_{m=1}^{M}.
$$

Packets arriving within the same scheduler tick or arrival bucket may be
batched.

For topology-changing runs:

```text
ledger entries must retain source lineage, target lineage, and edge lineage,
or provide an auditable aggregation key that can be resolved across refinement.
```

Lineage should not be dropped in runs that may refine, collapse, or reabsorb.
During refinement, packet transport must use the serialized lineage map. When
old port/column data exists, packets crossing old boundary edges should follow
the same old-port / old-column reassignment evidence used by the mechanical
expansion. Packet amount must be invariant unless an explicit arrival,
rollback, or cancellation event accounts for the difference.

Status:

```text
default compact representation for fixed topology;
lineage-bearing ledger required for LGRC-3.
```

### G.6 Uniform Proper-Time Inheritance

First-round LGRC-3 refinement should use uniform inheritance:

$$
\tau_{\mathrm{core}}=\tau_s,
\qquad
\tau_{\mathrm{satellite},b}=\tau_s
\quad(b=1,2,3).
$$

New internal edges receive an initial delay:

$$
\tau_{\mathrm{internal}}=\tau_0.
$$

The module inherits the parent's proper time at the refinement event time, not
at the last checkpoint. Any in-flight packet redirected into the module must
use the post-refinement lineage map and new internal delays only after the
expansion event.

Status:

```text
default for first LGRC-3 refinement tests;
column-offset or core-first policies remain future alternatives.
```

### G.7 Proper-Time Identity Threshold

First-round identity persistence should be calibrated to local causal
traversal time:

$$
\tau_{\mathrm{persist}}(B_{\mathrm{child}})
=
\kappa_{\mathrm{persist}}
\cdot
\mathrm{median}_{e\in\mathcal N(B_{\mathrm{child}})}\tau_e,
\qquad
\kappa_{\mathrm{persist}}\in[3,5].
$$

The neighborhood may be the child basin boundary, the child basin's past/future
cone, or another serialized local-basin policy. A global median may be used as
a fallback, but the local policy is preferred.

The first LGRC9V3 identity implementation should use either sink-local proper
time or lineage proper time and serialize the choice.

Status:

```text
default heuristic;
calibration open.
```

### G.8 GRC / LGRC Comparison Surfaces

Do not compare LGRC and GRC only by raw scheduler step. First-round comparison
should report:

1. mean proper time $\bar\tau$;
2. median proper time;
3. proper-time quantile bands;
4. event-count normalized curves;
5. matched causal event classes, such as pre/post expansion;
6. checkpoint surfaces only as observer/replay artifacts.

A useful global surface is:

$$
\bar\tau
=
\frac{1}{|V|}\sum_{i\in V}\tau_i.
$$

For spark, refinement, and identity comparisons, align by causal event class
rather than only by $\bar\tau$.

Status:

```text
default comparison policy for LGRC/GRC studies.
```

### G.9 Operational Lorentzian Structure

First-round LGRC remains operational:

```text
Lorentzian means causal-structure Lorentzian,
not tensor-signature Lorentzian.
```

The runtime establishes Lorentzian structure through:

1. directed past/future cones;
2. monotone local proper time;
3. causal delay;
4. lapse;
5. causal availability constraints;
6. event ordering.

It does not require an algebraic graph metric with signature $(-,+,+,+)$.

Status:

```text
default interpretation for this paper;
algebraic signature remains future work.
```

### G.10 Flux Basin Versus Causal Basin Core

LGRC should not immediately replace the synchronous flux basin. It should
distinguish:

| Basin notion | Meaning |
|---|---|
| Full flux basin $B_s^{\mathrm{flux}}$ | Nodes whose directed flux successor route reaches sink $s$. |
| Causal basin window | Nodes in the flux basin whose route to $s$ is reachable within a configured proper-time budget. |
| Causal basin core | Nodes that remain within repeated causal reachability windows over time. |

First-round definition:

$$
B_s^{\mathrm{causal}}(\Delta\tau)
=
\{\,i\in B_s^{\mathrm{flux}}
\mid
\tau_{\mathrm{path}}(i\to s \text{ along successor route})
\le
\Delta\tau\,\}.
$$

This should be read as a causally active basin core, not as destruction of the
full flux basin. Large basins may have valid members outside a short causal
window.

Status:

```text
derived in LGRC-0/LGRC-1;
causal availability affects updates in LGRC-2;
causal basin core affects identity persistence in LGRC-3.
```

### G.11 Collapse And Reabsorption Events

Collapse and reabsorption should be represented as causal lineage events.

A collapse or reabsorption payload should include:

```text
scheduler_event_index
node_proper_time or lineage proper time
causal_epoch
competing_sinks
selected_sink
losing_basin_mass
selected_basin_mass
transferred_coherence
budget_before
budget_after
budget_error
proper_time_transfer_policy
lineage_transfer
in_flight_packet_policy
```

Reabsorption may be represented either as explicit directed coherence packets
or as a basin-lineage transfer event that schedules budget-preserving packet
updates.

Status:

```text
default future LGRC-3 event contract;
not required for LGRC-0/LGRC-1.
```
